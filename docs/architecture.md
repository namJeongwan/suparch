# Architecture

Suparch separates data production from MCP serving.

```text
                 data pipeline
saved HTML / allowed fetch
          |
          v
  iHerb product parser
          |
          v
 normalized Product models
          |
          v
catalog-next.sqlite
 integrity_check + metadata
          |
          v
 versioned object storage
          |
          | startup download or mounted file
          v
read-only SQLite repository
          |
          v
stateless MCP server
```

## Runtime boundary

The public MCP process does not crawl iHerb or mutate its catalog. It opens an
immutable SQLite snapshot using `mode=ro` and serves deterministic retrieval
and calculation tools.

This keeps Hub deployments stateless and avoids losing writes when an
ephemeral container is restarted. It also allows the same database artifact to
run in a local stdio MCP process.

## Catalog publication

`SQLiteCatalogBuilder` creates the complete database in a temporary file,
runs `PRAGMA integrity_check`, vacuums it, and publishes it using an atomic
rename.

Production publication should use versioned artifacts:

```text
catalog-2026-07-16.sqlite
catalog-2026-07-16.sqlite.sha256
```

The MCP container can receive the database through either:

1. `SUPARCH_DB_PATH`: a mounted local path.
2. `SUPARCH_CATALOG_URL`: an HTTPS artifact downloaded at startup.

When a checksum is available, pass it through `SUPARCH_CATALOG_SHA256`.

## SQLite schema

- `products`: product identity, label serving, offer, and provenance.
- `product_ingredients`: active label rows and normalized quantities.
- `other_ingredients`: non-active ingredients in label order.
- `product_search`: FTS5 index for names, brands, ingredients, and forms.
- `metadata`: schema and catalog build information.

All numeric values are stored as decimal strings to avoid floating-point
rounding. MCP responses validate them as `Decimal`.

The current SQLite `user_version` is `2`. Catalogs generated by earlier schema
versions must be rebuilt from their normalized JSON source before deployment.

## Medical boundary

Suparch provides product facts and arithmetic:

- search labels;
- compare products;
- add user-supplied servings;
- identify duplicate active ingredients.

It does not infer a condition from symptoms, select a nutrient target, decide
whether an amount is safe, or prescribe a supplement. Domain-specific clients
and skills own those decisions.

## Upgrade path

SQLite remains the default while the runtime is read-heavy and the catalog is
published as snapshots. A PostgreSQL repository is only justified when the
system needs concurrent writers, user state, live offer updates, or large
history queries.
